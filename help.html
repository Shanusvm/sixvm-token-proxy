One proxy for all your AI · see every token, spend less on all of them.

SixVM Token Proxy sits between your AI agents and Anthropic (Claude), OpenAI (ChatGPT), and Google (Gemini). Your agents work exactly the same — every request passes through untouched — but everything gets measured, and a stack of automatic money-savers works quietly in the background.

Your agents ──► http://localhost:8787 ──► Claude · ChatGPT · Gemini │ ├── measures every request (tokens, cost, speed) └── saves money (caching · routing · budgets · retries)
Your API keys stay on your machine. Keys live only in a local .env file, are never uploaded anywhere, and are only ever sent to the AI provider they belong to. Add them on the Setup page — each key is verified with its provider before saving.
Your data stays yours — nothing is reported to anyone. Usage records live in a local database file next to the proxy, or (optionally) in your own free Supabase project that only you control. Prompt and answer content is never stored to disk at all — the optional Request explorer keeps the last 100 in memory only, on your machine.

Getting started (about 3 minutes)

  1. Install Node.js

    Version 22 or newer from nodejs.org. Check with node --version.

  2. Download & start the proxy 
    git clone https://github.com/YOUR-USERNAME/sixvm-token-proxy.git
cd sixvm-token-proxy
npm install
npm start
  3. Add your API keys

    Open http://localhost:8787/setup and paste the keys for the AIs you use: Anthropic, OpenAI, and/or Google Gemini. Any one is enough; add the rest later.

  4. Test it

    Open the dashboard, pick an AI in the dropdown, click Send test. The reply appears next to the button and the request lands in the table instantly. Usage is logged to a local database file automatically — nothing to configure.

Connecting your agents

Change one URL per agent. Add the two tag headers so the dashboard can tell your agents apart.

Claude · Anthropic SDK

const client = new Anthropic({
  apiKey: "not-needed",              // the proxy holds the real key
  baseURL: "http://localhost:8787",
  defaultHeaders: {
    "x-sixvm-client": "my-agent",    // who is calling
    "x-sixvm-task": "summarize",     // what kind of work
  },
});

ChatGPT · OpenAI SDK

const openai = new OpenAI({
  apiKey: "not-needed",
  baseURL: "http://localhost:8787/v1",
  defaultHeaders: { "x-sixvm-client": "my-agent", "x-sixvm-task": "chat" },
});

Environment variables (Claude Code, many tools)

ANTHROPIC_BASE_URL=http://localhost:8787
OPENAI_BASE_URL=http://localhost:8787/v1

Gemini · raw endpoint

POST http://localhost:8787/v1beta/models/gemini-2.5-flash:generateContent

Streaming works exactly like the real APIs, and other endpoints (count_tokens, models, …) are forwarded too.

Optional: your own cloud database (Supabase)

By default the proxy logs usage to a local file — zero setup. If you'd like your usage data in the cloud (it survives reinstalls, and you can query it from anywhere), connect your own free Supabase project. The data goes only to your project; nobody else can see it.

  1. Create a free Supabase project

    Sign up at supabase.com → New project → any name, free plan. Wait a minute while it provisions.

  2. Copy your two values

    In your project: Project Settings → API keys. Copy the Project URL (looks like https://xxxxx.supabase.co) and the service_role secret key.

  3. Paste them into Setup

    Open Setup → section 3 · Your usage database, paste both values, click Test & Save. The connection is verified immediately.

  4. Run the table SQL once

    Setup will show you a block of SQL. Copy it, go to Supabase's SQL Editor (left menu → New query), paste, click Run. Back in Setup, click Test & Save once more — done. New usage now flows to your Supabase.

Note: switching to Supabase starts a fresh history there — rows already in your local file stay in the local file.

Frequently asked questions

Is my API key shared with anyone?

No. Keys live in a local .env file, excluded from Git, never transmitted to SixVM or anyone else — only to the AI provider each key belongs to, exactly as your agents already did.

Does SixVM (or anyone) see my usage?

No. Nothing is reported anywhere. Usage records stay in your local database file, or in your own Supabase project if you connected one — a database only you control.

Whose AI account pays for requests?

Yours — the accounts behind the keys you enter at Setup. The proxy adds no fees.

Does the proxy slow my agents down?

Only a few milliseconds — and often it makes them faster: cached repeat answers return in ~100 ms instead of seconds, and retries hide provider hiccups.

Where is my usage data stored?

By default in a local database file (data/usage.db) on your machine, automatically. Optionally in your own free Supabase project — see "Optional: your own cloud database" above. Either way, if the database is ever unavailable, requests still pass through fine.

Can I use it with Claude Code?

Yes — set ANTHROPIC_BASE_URL=http://localhost:8787. One caveat: if your Claude Code runs on a Pro/Max subscription, routing through the proxy bills usage to your API key instead, so most people track only their agents.

Why did a request use a different model than my agent asked for?

Either a routing rule, the auto-router (downgrading a simple task to a cheaper model), or failover (the requested provider was down). Every switch is listed in the dashboard's "Auto-router activity" table with the reason. Turn any of these off in Setup.

Can I run it on a server instead of my PC?

Yes, any machine with Node.js 22+. But if the server is reachable from the internet, put the proxy behind a firewall or VPN — anyone who can reach it can spend your AI credits. Setup changes are only accepted from the machine it runs on.

The est. cost numbers look wrong

Per-token prices live in the PRICING table at the top of server.js. Prices change — check the providers' pricing pages and update the numbers there.
