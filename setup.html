SixVM Token Proxy · Setup

Two steps and you're running. Keys are stored only in the local .env file on this machine · never sent anywhere else. New here? Read the 5-minute guide.

Claude: checking… ChatGPT: checking… Gemini: checking… Local database: checking…

1 · Anthropic (Claude) API key

Get one at console.anthropic.com → API keys. Every key is verified with its provider before saving (no tokens are used). Keys are stored only on this machine.

2 · OpenAI & Gemini keys (optional · add the providers your agents use)

Agents point at this proxy for each provider: OpenAI via OPENAI_BASE_URL, Gemini via the SDK's endpoint setting. Usage from all providers lands on the same dashboard.

3 · Your usage database (optional · your own free Supabase)

Without this, usage is stored in a local file next to the proxy · zero setup, works fine. Connect your own free Supabase project if you want your data in the cloud: it survives reinstalls and you can query it from anywhere. Your data goes only to your project · nobody else can see it.

How to set it up (about 5 minutes, free):
1 · Create an account at supabase.com and click New project (any name, the free plan is enough)
2 · Wait a minute for it to provision, then open Project Settings → API keys
3 · Copy the Project URL and the service_role secret key into the boxes below
4 · Click Test & Save · the SQL to create the table appears below: copy it, paste into Supabase's SQL Editor (left menu → SQL Editor → New query), press Run once, then click Test & Save again

One more step: run this SQL once in your Supabase SQL Editor, then click Test & Save again:

create table if not exists token_usage (
  id bigint generated always as identity primary key,
  created_at timestamptz not null default now(),
  client text,
  task_type text,
  model text,
  input_tokens int,
  output_tokens int,
  cache_read_tokens int,
  cache_write_tokens int,
  cache_hit boolean default false,
  est_cost numeric,
  request_id text,
  latency_ms int
);

create index if not exists token_usage_created_at_idx on token_usage (created_at);
create index if not exists token_usage_client_idx on token_usage (client);

create or replace view token_usage_daily as
select
  date_trunc('day', created_at) as day,
  client,
  model,
  count(*) as requests,
  sum(input_tokens) as input_tokens,
  sum(output_tokens) as output_tokens,
  sum(cache_read_tokens) as cache_read_tokens,
  sum(cache_write_tokens) as cache_write_tokens,
  avg(est_cost) as avg_est_cost,
  avg(latency_ms) as avg_latency_ms
from token_usage
group by 1, 2, 3;

4 · Optimizations & limits (applies instantly, no restart)

Automatic money-savers and safety limits. Stored locally in config.json.

Adds Anthropic's cache marker to large system prompts (5,000+ characters), so repeated prompt content is billed at ~10% of the normal price. Requests are otherwise unchanged.

If the exact same request arrives again, the proxy answers from memory · zero tokens spent. Answers can be up to the cache age old, so keep it off for time-sensitive agents.

A client that reaches its monthly budget gets a clear "budget exceeded" error instead of silently spending more.

You get a message when the day's total spend crosses this line, and whenever a client hits its budget.

With the explorer on, click any row on the dashboard to see the exact prompt and answer. Kept in memory on this machine only · never written to disk, never reported anywhere.

5 · Advanced token savers (per task tag)

These act on the x-sixvm-task tag your agents send. Leave empty to change nothing.

Send simple tasks to a cheaper model automatically. Haiku (claude-haiku-4-5) is 5× cheaper than Sonnet and 25× cheaper than Fable-class models. Use * as the task to route everything.

Output tokens cost 5× more than input. This caps what any single request may ask for.

Lower effort = the model thinks and writes less, sharply cutting tokens on routine work. Applied only on models that support it, and only when the agent didn't set an effort itself.

6 · Auto-router (picks the model automatically)

The proxy judges each request's difficulty and picks the cheapest model that can handle it: simple → Haiku, medium → Sonnet, hard → whatever was requested. It never switches model mid-conversation (that would forfeit the cache discount) and manual routing rules from section 5 always win.

7 · Reliability (retries & backup model)

When a provider is overloaded or rate-limits, the proxy retries automatically with short waits before giving up · your agents never see the hiccup.

Failover answers arrive in Claude's response format, so agents keep working without changes. Requests with tools or images are never failed over.
